# 任務列表：[FEATURE NAME]

**輸入**: 來自 `/specs/[###-feature-name]/` 的設計文件

**前提條件**: plan.md (必填), spec.md (使用者故事必填), research.md, data-model.md, contracts/

**測試**: 下方的範例包含測試任務。測試為選填項目——僅在功能規格書中明確要求時才需包含。

**組織結構**: 任務依使用者故事 (User Story) 進行分組，以便獨立實作和測試每個故事。

## 格式：`[ID] [P?] [Story] 描述`

- **[P]**: 代表可並行執行（不同檔案，無相依關係）
- **[Story]**: 該任務所屬的使用者故事（例如：US1, US2, US3）
- 描述中需包含確切的檔案路徑

## 路徑規範

- **單一專案**: 專案根目錄下的 `src/` 與 `tests/`
- **網頁 App**: `backend/src/` 與 `frontend/src/`
- **行動裝置**: `api/src/`、`ios/src/` 或 `android/src/`
- 下方範例路徑假設為單一專案——請根據 plan.md 結構進行調整

## 依賴關係與執行順序

### 階段依賴關係

- **專案設定 (Phase 1)**: 無相依性——可立即開始
- **基礎建設 (Phase 2)**: 依賴設定階段完成——會「阻擋」所有使用者故事的實作
- **使用者故事 (Phase 3+)**: 均依賴基礎建設階段的完成
  - 各使用者故事隨後可並行進行（若有人力分工）
  - 或按優先順序（P1 → P2 → P3）依序實作
- **收尾階段 (Final Phase)**: 依賴於所有選定實作的使用者故事完成

### 使用者故事相依性

- **使用者故事 1 (P1)**: 可在基礎建設（Phase 2）完成後開始——不依賴其他故事
- **使用者故事 2 (P2)**: 可在基礎建設（Phase 2）完成後開始——可能與 US1 整合，但必須能獨立測試
- **使用者故事 3 (P3)**: 可在基礎建設（Phase 2）完成後開始——可能與 US1/US2 整合，但必須能獨立測試

### 每個使用者故事內部

- 測試（如果包含）必須先編寫，並在實作前確認失敗
- 模型（Models）先於服務（Services）
- 服務（Services）先於 API 端點（Endpoints）
- 核心實作先於整合
- 故事必須完全完成後，再移至下一個優先順序的故事

### 並行機會

- 所有標記為 [P] 的設定任務可以並行執行
- 所有標記為 [P] 的基礎建設任務可以並行執行（在 Phase 2 內）
- 一旦基礎建設階段完成，所有使用者故事都可以並行開始（若團隊產能允許）
- 某個故事中所有標記為 [P] 的測試可以並行執行
- 某個故事中標記為 [P] 的模型可以並行執行
- 不同的團隊成員可以並行開發不同的使用者故事

## 實作策略

### MVP 優先 (僅限使用者故事 1)

1. 完成 Phase 1: 專案設定
2. 完成 Phase 2: 基礎建設 (關鍵——阻擋所有故事)
3. 完成 Phase 3: 使用者故事 1
4. **停止並驗證**: 獨立測試使用者故事 1
5. 若就緒則進行部署/演示

### 增量交付

1. 完成設定 + 基礎建設 → 基礎建設就緒
2. 加入使用者故事 1 → 獨立測試 → 部署/演示 (MVP!)
3. 加入使用者故事 2 → 獨立測試 → 部署/演示
4. 加入使用者故事 3 → 獨立測試 → 部署/演示
5. 每個故事在不破壞前述故事的前提下增加價值

### 團隊並行策略

多人開發時：

1. 團隊共同完成設定 + 基礎建設
2. 基礎建設完成後：
   - 開發人員 A: 使用者故事 1
   - 開發人員 B: 使用者故事 2
   - 開發人員 C: 使用者故事 3
3. 各故事獨立完成與整合

## 備註

- [P] 任務 = 不同的檔案，無相依關係
- [Story] 標籤將任務映射到特定的使用者故事以進行追蹤
- 每個故事都應該能獨立完成和獨立測試
- 在實作前驗證測試是否失敗
- 在每個任務或邏輯群組完成後進行 commit
- 在任何檢查點停止，以獨立驗證故事
- 避免：模糊的任務、同檔案衝突、破壞獨立性的跨故事相依關係
